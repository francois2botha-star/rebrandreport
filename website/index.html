What it does

The website handles business. The app handles operations.

People request projects, workspaces, access, and customisation here. Approved work then lives inside a private RolloutHQ workspace instance.

01

Project intake

Capture project quotes, new workspace requests, user access requests, and early scoping details before anyone receives app access.

02

Workspace instances

Each client programme gets a controlled workspace with its own users, roles, brand context, project types, files, reports, and support paths.

03

Support intelligence

Feature requests and report ideas become product signals, while client administrators approve changes before they reach implementation.

Workspace model

Built for consultants, agencies, operators, and client delivery teams.

RolloutHQ stays generic and resellable. A signage rollout can be one workspace. A facilities programme, service delivery queue, public project, or supplier network can be another.

  • Workspace administrator controls client operations.
  • Client administrator approves client-side changes.
  • Site contacts and suppliers see only their assigned project context.
  • RolloutHQ support gathers useful improvement requests across workspaces.

Request paths

Every business request starts outside the workspace.

That separation keeps the app clean: only approved users and active project parties enter the operational instance.

For clients who need a quote, estimate, or scoped project added to an existing workspace.

New workspace

For agencies or companies that want their own branded project environment.

Access request

For users who need to join a workspace as a client, site contact, supplier, or viewer.

Customisation

For reports, workflow stages, dashboards, templates, permissions, or support ideas.
